# Benchmark: Codex Native Subagents vs. Subagent Router

## TL;DR

I benchmarked three ways to build the same small distributed system:

1. **Codex 5.5 Medium alone**
2. **Codex 5.5 Medium with native Codex subagents**
3. **Codex 5.5 Medium using Subagent Router with DeepSeek V4 Flash/Pro subagents**

The task was to implement a Python distributed task queue from scratch using only the standard library, then audit it for priority-queue races and socket-level memory leaks.

The outcome was not a single winner across every dimension:

| Category                                   | Best result                |
| ------------------------------------------ | -------------------------- |
| Fastest run                                | Native Codex subagents     |
| Lowest visible Codex limit usage           | Codex alone                |
| Strongest core queue-state model           | Native Codex subagents     |
| Most operationally hardened implementation | Subagent Router + DeepSeek |
| Weakest implementation                     | Codex alone                |

The practical conclusion:

> Codex alone produced a quick prototype, but the weakest implementation. Native Codex subagents handled the core queue-state model best. Subagent Router produced the most operationally hardened implementation and used less of the visible Codex 5-hour limit than the native-subagent run.

This is not a claim that Subagent Router “writes better code.” Subagent Router is not a model. It is a routing, profile, observability, provider-choice, and budget-control layer. In this benchmark, the code-producing system was **Codex 5.5 Medium as parent/coordinator plus DeepSeek V4 Flash and DeepSeek V4 Pro as routed worker/reviewer models**.

## Why this benchmark matters

Subagent Router has two economic goals.

The first is **subscription-limit extension**. If useful work can be delegated outside Codex while consuming less of the Codex 5-hour and weekly limits, a fixed subscription lasts longer.

The second is **API-spend reduction**. In API-billed workflows, a premium OpenAI model can stay in the parent/orchestrator role while cheaper or local models handle eligible exploration, implementation, review, and remediation.

This benchmark mainly tests the first goal. It also includes a rough API-equivalent estimate for the second.

## Task prompt

Each run received the same benchmark task:

> Implement a Python-based distributed task queue from scratch using only standard libraries. It must support task prioritization and worker heartbeats. Once implemented, perform a rigorous security and performance audit of the code, specifically checking for race conditions in the priority queue and socket-level memory leaks. Use subagents where applicable.

This task is small enough to review manually, but it is not a toy completion prompt. It involves:

* concurrent state transitions
* priority ordering
* worker leases and heartbeats
* stale-worker recovery
* socket protocol boundaries
* connection cleanup
* memory-retention behavior
* audit quality
* test coverage

## Methodology

For each run, I recorded:

* final source files
* reported test results
* runtime
* visible Codex session token summary
* 5-hour and weekly limit movement
* Subagent Router external cost where available
* manual code-review findings after the run

I then compared the generated code against both the agents’ own tests and additional adversarial cases found during review.

This is an engineering benchmark, not a formal academic evaluation. The test suites were generated by the agents and were not identical. That is why the final verdict is based on both test results and source-level inspection.

## Findings by approach

### Codex alone: quick prototype, weakest implementation

The no-subagent run finished quickly and consumed the least visible 5-hour capacity. It also produced the shallowest implementation.

The gaps were mostly around production-adjacent behavior:

* no comparable connection-cap story
* weaker socket lifecycle cleanup
* weaker retention/pruning story
* weaker heartbeat failure resilience
* fewer protocol-abuse checks
* less security/performance audit depth

This baseline is important because it shows what Codex 5.5 Medium produced without delegated review pressure. On this benchmark, that output was clearly the lowest-quality result.

### Native Codex subagents: strongest core queue model

Native subagents made the best core queue-state decisions.

Two choices stood out:

1. Track multiple in-flight tasks per worker.
2. Requeue a task if the broker cannot write the task response to the worker socket.

Those are not style choices. They prevent stuck tasks and task-loss-like behavior.

Native was also the fastest run. Its weakness was the surrounding operational envelope: connection pressure, protocol abuse, task retention, and heartbeat-loop failure handling were less complete than in the Subagent Router result.

### Subagent Router + DeepSeek: strongest operational hardening

The Subagent Router run produced the most defensive implementation overall.

It had broader safeguards around:

* max connection caps
* active socket tracking
* forced socket cleanup on shutdown
* partial-connect cleanup
* invalid UTF-8
* JSON depth, key, and item limits
* max task limits
* completed-task pruning
* worker deregistration cleanup
* heartbeat exception resilience
* socket/protocol regression coverage

That is why I rank it as the most operationally hardened implementation, even though native Codex subagents still handled the core active-task model better.

## Code-level findings

### 1. Multiple active tasks per worker

**Winner: native Codex subagents**

Native used a set of in-progress task IDs per worker:

```python
self.in_progress: dict[str, set[str]] = {}
self.in_progress.setdefault(worker_id, set()).add(task.id)
```

When a worker goes stale, all active tasks for that worker can be requeued.

The Subagent Router implementation tracked only one active task per worker. If the same worker reserved two tasks, the second reservation could overwrite the first active task pointer. If the worker then went stale, only the latest task was requeued.

Manual adversarial check:

```text
1_2 router cost profile: task2 -> QUEUED, task1 -> RESERVED
2 no subagents:          task2 -> queued, task1 -> running
3 native subagents:      task1 -> queued, task2 -> queued
```

This is the strongest technical argument for native subagents in this benchmark.

### 2. Rollback after response-write failure

**Winner: native Codex subagents**

Native explicitly requeued a task if the broker failed to write the fetch response to the worker. That prevents a task from staying marked as running when the worker never actually received it.

The router implementation caught send failure, but recovery depended on stale-worker timeout. That is weaker than immediate rollback.

Fix for the router version:

* if reservation succeeds and response write fails, requeue immediately
* remove the task from the worker’s active set
* add a regression test for this path

### 3. Connection pressure

**Winner: Subagent Router + DeepSeek**

The router implementation had the clearest resource-pressure defense:

* max connection cap
* active socket tracking
* server-at-capacity response
* connection count decrement on close
* forced socket close on shutdown
* partial-connect cleanup tests

Native and no-subagent versions used thread-per-connection models without an equivalent connection cap.

### 4. Protocol hardening

**Winner: Subagent Router + DeepSeek**

The router implementation checked more protocol abuse cases:

* max line size
* invalid UTF-8
* non-object JSON
* excessive nesting
* excessive keys
* excessive array items

For a socket-facing service, those checks matter. A byte cap alone does not protect against every memory or CPU abuse case.

Remaining gap: add an outgoing response size cap.

### 5. Heartbeat resilience

**Winner: Subagent Router + DeepSeek**

The router worker caught heartbeat exceptions so a transient coordinator or socket error would not permanently kill the heartbeat thread.

The native and no-subagent versions were weaker here. If the heartbeat thread dies, the worker loop may still be alive while the broker eventually marks it stale.

### 6. Retention and memory growth

**Winner: Subagent Router + DeepSeek**

The router implementation had max task limits and pruning. Native retained tasks in memory without a comparable retention policy. The no-subagent version also had a weaker retention story.

## Subscription-limit behavior

For subscription users, the practical question is:

> How much of the Codex 5-hour and weekly limit did this task consume for the quality of result produced?

In this benchmark:

| Implementation   | 5h movement | Quality summary                            |
| ---------------- | ----------- | ------------------------------------------ |
| Codex alone      | ~1 point    | weakest implementation                     |
| Native subagents | ~5 points   | best core queue model                      |
| Subagent Router  | ~3 points   | most operationally hardened implementation |

The no-subagent run spent the least visible 5-hour capacity, but produced the weakest code. The native run was fastest and had the best core queue model, but moved the 5-hour limit the most. The router run landed in the middle on 5-hour movement and produced the most defensive implementation.

That is the useful signal:

> Subagent Router produced stronger code than the no-subagent baseline while consuming less of the visible 5-hour limit than native Codex subagents in this run.

## Rankings

These rankings are based on the criteria in this benchmark: queue correctness, socket/resource safety, protocol hardening, test coverage, and usage-limit movement. They are not a general model leaderboard.

### Most operationally hardened source code

1. `1_2_codex_deepseek_cost_optimization_profile`
2. `3_codex_native_subagents`
3. `2_codex_no_subagents`

The router implementation had the broadest hardening across connection pressure, protocol abuse, cleanup, retention, and heartbeat resilience.

### Best core queue correctness

1. `3_codex_native_subagents`
2. `1_2_codex_deepseek_cost_optimization_profile`
3. `2_codex_no_subagents`

Native handled multiple active tasks per worker and fetch-response rollback better.

### Best limit-extension signal

1. `1_2_codex_deepseek_cost_optimization_profile`

The router run produced stronger code than the no-subagent baseline while moving the 5-hour limit less than the native-subagent run.

### Fastest implementation

1. `3_codex_native_subagents`
2. `2_codex_no_subagents`
3. `1_2_codex_deepseek_cost_optimization_profile`

Speed matters, but this prompt explicitly required a rigorous security/performance audit.

## What I would fix next

For the Subagent Router implementation:

1. Replace single `active_task_id` with `active_task_ids: set[str]`.
2. Requeue all active tasks on stale worker or deregistration.
3. Clear only the completed or failed task from the worker’s active set.
4. Roll back reservation immediately if task response write fails.
5. Add an outgoing response size cap.
6. Add a final reviewer pass when agent capacity allows.
7. Add adversarial tests for same-worker multiple reservations and response-write failure after reservation.

For the native-subagent implementation:

1. Add max connection limits.
2. Add active socket tracking and forced cleanup.
3. Add completed/failed task retention or pruning.
4. Catch heartbeat-loop exceptions.
5. Add JSON depth/key/item hardening.
6. Expand tests beyond the current 9 tests.

## What this says about Subagent Router

The benchmark did not prove that external cheap subagents are always better than native subagents.

It showed something narrower:

* Codex alone can produce a quick prototype, but quality suffered on this task.
* Native Codex subagents have strong integrated coordination and handled the core queue model well.
* Subagent Router can make a Codex parent use cheaper external models for substantial implementation and review work.
* Router profiles matter. A single global instruction policy is not enough.

During benchmarking, three distinct profile needs became clear:

| Profile             | Purpose                                                                                                                                    |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `orchestrator`      | Keep Codex in broad control. Good for quality, weaker for limit extension.                                                                 |
| `deep-delegation`   | Push exploration, implementation, review, and remediation aggressively to router agents.                                                   |
| `cost-optimization` | Minimize parent Codex work with stricter output contracts, retry caps, selective reviewer usage, and tighter limits on parent local edits. |

The final cost-optimization run got closest to the intended behavior: the parent mostly coordinated and verified instead of implementing.

It is still imperfect. The run needed bounded local inspection, hit an agent-capacity limit before a final reviewer pass, and produced code with some adversarial queue-state gaps.

Product positioning from this benchmark:

> Subagent Router is not a replacement for native Codex subagents, and it is not a code-generation model. It is a configurable offload layer for using local, cheaper, or alternative backends when users want provider choice, privacy, routing policy, external review, longer Codex subscription usage, or lower OpenAI API spend.

## Limitations

This is an exploratory benchmark.

Limitations:

* One task type: a Python distributed task queue.
* One parent model configuration: Codex 5.5 Medium.
* Test suites were agent-generated and not identical.
* Parent token summaries came from session reports, not independent telemetry.
* Native subagent token usage is not separately visible in the parent-window token summary.
* API-equivalent estimates are approximate visible-token estimates, not actual Plus subscription billing.
* Human post-run code review influenced the final interpretation.
* The benchmark measures implementation quality and session-limit behavior, not long-term maintainability.

The next round should use a fixed adversarial test suite that every implementation must run.
